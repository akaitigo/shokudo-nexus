# PRD: shokudo-nexus

**作成者**: Ryusei | **日付**: 2026-03-29 | **ステータス**: ドラフト
**アイデアID**: 549 | **ドメイン**: 子ども食堂 | **難易度**: Medium

## 問題定義

全国に7,000箇所以上ある子ども食堂は、食品の調達・在庫管理を手作業で行っている。一方でスーパーや農家から大量の余剰食品が廃棄されている。消費期限の短い生鮮食品を適切なタイミングで必要な食堂にマッチングする仕組みがなく、食品ロスと食堂の運営非効率が同時に発生している。

shokudo-nexus は、余剰食品と子ども食堂のリアルタイム需要を gRPC でマッチングするサプライチェーン管理プラットフォームである。食品の「消費期限」と食堂の「献立予定」をベクトル検索でマッチングし、地域内での食の循環を最適化する。

## 目標と成功指標

| 目標 | 指標 | ターゲット |
|------|------|----------|
| 食品マッチング効率化 | 余剰食品の登録から食堂への引取りまでの平均時間 | 24時間以内 |
| 食品廃棄削減 | 登録された余剰食品のうちマッチングされた割合 | 70%以上 |
| 拠点間連携促進 | 月あたりの拠点間食材融通リクエスト数 | 50件以上 |
| 自治体レポート自動化 | レポート生成にかかる時間 | 手動の1/10以下 |
| ユーザー定着 | 月間アクティブ食堂数 / 登録食堂数 | 60%以上 |

## スコープ外

- 配車・物流の最適化（MVP後のフェーズで検討）
- 決済・課金機能
- 食品安全基準の法的コンプライアンス認証
- モバイルネイティブアプリ（Webレスポンシブで対応）
- ベクトル検索によるマッチング最適化（MVP後）

## ユーザーストーリー

- **子ども食堂運営者**として、今週の献立に必要な食材をリクエストしたい。なぜなら、必要な食材を効率的に調達し、廃棄を減らしたいから。
- **食品ドナー（スーパー・農家）**として、余剰食品をリアルタイムで登録したい。なぜなら、廃棄前に必要な場所に届けたいから。
- **子ども食堂運営者**として、近隣食堂と食材を融通し合いたい。なぜなら、余った食材を無駄にせず、足りない食材を補いたいから。
- **自治体担当者**として、地域の食品寄付実績レポートを自動生成したい。なぜなら、補助金申請や政策立案のデータが必要だから。
- **子ども食堂ネットワーク管理者**として、複数拠点の在庫状況を一覧で把握したい。なぜなら、ネットワーク全体の食材過不足を調整したいから。

## 技術要件

### 技術スタック

- **バックエンドAPI**: Go (gRPC)
- **フロントエンド**: TypeScript / React
- **データベース**: Firestore (GCP)
- **インフラ**: GCP Cloud Run
- **Proto管理**: buf

### アーキテクチャ

モノレポ構成（backend/ + frontend/ + proto/）。バックエンドは Go で gRPC サービスを実装し、フロントエンドは gRPC-Web 経由で通信する。Proto定義は proto/ に集約し、buf で管理する。

```
[React SPA] --gRPC-Web--> [Go gRPC Server] ---> [Firestore]
                              |
                         [Cloud Run]
```

### 非機能要件

- パフォーマンス: API レスポンス 200ms 以内（p95）
- セキュリティ: 認証は Firebase Auth、通信は全て TLS
- スケーラビリティ: 同時接続100食堂、1日あたり1,000件の食品登録

## 競合分析

| 競合 | URL | ギャップ |
|------|-----|---------|
| こども食堂ネットワーク | https://musubie.org/ | 情報ポータルのみ。食品管理SaaS機能なし |
| TABETE | https://tabete.me/ | 一般消費者向けフードロス削減。子ども食堂特化ではない |
| セカンドハーベスト | https://2hj.org/ | フードバンク運営。技術プラットフォームではない |

**差別化ポイント**: 食品サプライチェーン管理 x 生鮮食品消費期限最適化で子ども食堂の運営効率化を技術的に支援。子ども食堂向けの食品管理SaaSは現在皆無。

## 技術リスク

| リスク | 発生可能性 | 影響 | 軽減策 |
|--------|----------|------|--------|
| gRPC-Web の互換性問題 | 中 | 高 | Envoy proxy or grpc-web ライブラリで対応。REST fallback も検討 |
| Firestore のクエリ制限 | 中 | 中 | 複合インデックス設計を事前に行う。必要に応じて BigQuery 連携 |
| リアルタイム通知の遅延 | 低 | 中 | Server Streaming RPC で実装。必要に応じて Pub/Sub 導入 |
| 消費期限データの標準化 | 中 | 高 | 入力バリデーションの厳格化。ISO 8601 日付形式で統一 |

## マイルストーン

### Phase 0: MVP（2週間）

1. **プロジェクト基盤**（2日）: CI/CD、リンター、テストフレームワーク、Proto定義
2. **コアデータモデル**（3日）: 食品、食堂、在庫のFirestoreスキーマ設計、gRPC サービス定義
3. **食品在庫管理API**（3日）: 余剰食品の登録・一覧・消費期限管理の gRPC サービス
4. **拠点間融通機能**（3日）: 食材リクエスト・マッチングの gRPC サービス
5. **フロントエンドMVP**（3日）: 食品登録画面、在庫一覧、融通リクエスト画面

### Phase 1: レポート機能（1週間）

- 寄付履歴の可視化ダッシュボード
- 自治体向けPDFレポート自動生成

### Phase 2: マッチング最適化（2週間）

- 消費期限 x 献立予定のベクトル検索マッチング
- 通知・アラート機能

## マネタイズ

- **フリーミアム**: 基本機能は無料（子ども食堂の公益性を考慮）
- **自治体向けプレミアム**: レポート自動生成、データ分析ダッシュボード
- **企業スポンサーシップ**: 食品ドナー企業の社会貢献PR連携
