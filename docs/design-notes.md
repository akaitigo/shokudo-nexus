# 設計ノート

実装上の設計判断とその根拠・許容リスクを記録する。ADR ほど重くない、コードに紐づく
判断を対象とする。

## 消費期限ステータスのオンザフライ更新（ベストエフォート）

### 概要

`GetFoodItem` / `ListFoodItems` は、`available` かつ消費期限を過ぎたアイテムを検出すると、
その場でステータスを `expired` に更新する（`backend/internal/service/food_inventory.go`）。

- `GetFoodItem`: 更新失敗時はエラーを返す（単一アイテムの整合性を優先）。
- `ListFoodItems`: 更新失敗時は**警告ログのみ**でリクエストは成功させる（ベストエフォート）。

### 判断根拠

- **一覧応答のレイテンシを優先**: 一覧取得は複数アイテムを走査する。各アイテムの期限切れ
  永続化を同期的にブロックすると、Firestore への書き込み回数分だけ応答が遅延する。
  期限切れの永続化は「次に誰かが読んだとき」に反映されれば十分であり、一覧表示の速度を
  優先してベストエフォートとした。
- **表示は常に正しい**: 期限切れ判定はレスポンス生成時に毎回オンザフライで行う
  （`FoodItem.IsExpired(now)`）。DB 上のステータス永続化が遅れても、クライアントに返る
  ステータスは常に最新の判定結果になる。

### 許容リスク

- 永続化に失敗したアイテムは DB 上 `available` のまま残るが、次回の `Get`/`List` で再び
  更新が試行される（自己修復的）。
- 承認処理（`RespondToFusionRequest`）はトランザクション内で `available` のみを承認対象と
  するため、DB 上のステータスが一時的に古くても、期限切れアイテムが誤って予約されることは
  ない（期限切れ＝`available` 以外なので、期限が切れていれば `IsExpired` によりオンザフライで
  `expired` 化され、承認前チェックで弾かれる）。
- リトライ機構・管理者通知は MVP のスコープ外とした。恒常的に書き込みが失敗する状況
  （Firestore 障害）は警告ログとモニタリングで検知する想定。

### 検討した代替案

- **同期更新（ブロッキング）**: 一覧レイテンシが悪化するため不採用。`GetFoodItem` は
  単一アイテムのため同期更新を採用している。
- **バックグラウンドジョブ（スケジューラで一括 expired 化）**: MVP では過剰。将来的に
  アイテム数が増えオンザフライ更新のコストが問題化した場合の選択肢として残す。

## 一覧のカーソルページネーションと全順序保証

`ListFoodItems` / `ListFusionRequests` のリポジトリ実装は、`created_at` 降順に加えて
ドキュメントID (`__name__`) を第2ソートキーとして明示的に指定する
（`backend/internal/repository/`）。

- `created_at` は複数ドキュメントで重複しうるため、単独では全順序にならず、カーソル
  （`StartAfter`）ベースのページネーションで重複・欠落が生じうる。
- ドキュメントIDを併用して全順序を保証することで、一覧取得中にステータス更新（期限切れ
  反映）が並行してもページトークンがずれない。`created_at` と `__name__` はステータス更新で
  変化しない不変キーである。
- この複合ソートに対応する Firestore 複合インデックスは `firestore.indexes.json` に定義する。

## 承認処理の排他制御（Firestore トランザクション）

`RespondToFusionRequest` の承認は Firestore トランザクション
（`repository.FirestoreApprovalRunner`）で在庫予約と状態遷移をアトミックに行う。

- プロセスローカルな `sync.Mutex` では Cloud Run のスケールアウト時に二重予約を防げないため、
  Firestore の楽観的並行制御に移行した。
- 同一在庫への並行承認は、一方がコミットするともう一方は競合を検知して再試行し、再試行時に
  `reserved` を読み取って `FailedPrecondition` で失敗する。結果として在庫はちょうど1回だけ
  予約される。
